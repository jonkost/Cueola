1. ✅ **Firestore offline persistence + write-failure semantics audit** (small, high) *(Done 2026-07-11.)*
   Both pages now use `initializeFirestore` + `persistentLocalCache({tabManager: persistentMultipleTabManager()})` with a memory-cache fallback. Semantics audit: the main session listener passes `includeMetadataChanges: true` (without it, an idle reconnect is metadata-only and the RECONNECTING chip never clears) and only claims "connected" on a server snapshot (`!fromCache`); `hasPendingWrites`/queued batches show as saving. `reportCloudWriteFailure` treats `unavailable`/`deadline-exceeded`/`!navigator.onLine` as the quiet reconnecting state — logged once, no error dot, no toast; real failures (rules, invalid data) still toast. Verified in the preview, all four layers: plain writes queue offline (REST-confirmed invisible on the server) and flush on restore; the rundown-transaction path under an `unavailable` outage produces zero toasts, saving dot, one log line, and the queue drains to the server on restore; a reload boots the show from IndexedDB (first snapshot `fromCache:true` with full data); app + dashboard tabs coexist with zero persistence errors.

## Phase 2 — Show-day armor
*A frozen output announces itself, a fat-fingered delete is undoable, a clobbered session doc is restorable, a reload on dead venue Wi-Fi still boots the show.*

1. ✅ **Output-window watchdog heartbeat with auto-recover + preflight row** (small, high) *(Done 2026-07-11.)*
   The output window now emits an unsolicited `beat` every 2 s carrying an rAF-freshness flag (`raf:false` = event loop alive but frames not painting — logged once when a video cue is live there; no beats at all = frozen/crashed). The controller pings every 2 s and declares an output dead after 5 s without a beat: pulsing **red dot** in the outputs panel, red state on the toolbar button, **one** toast, one show-log error (transition-gated, so no repeat spam). A beat after death auto-heals: recovery toast/log + `resendActiveToOutput` (which now also re-pushes the **keyer** state — a rebooted output loses its WebGL key) + `applyOutputSink`. Critical behavior change baked in: the old code re-sent the active cue **on every pong** — with a 2 s heartbeat that would have re-issued `play` constantly; resend now happens only on `ready` (fresh window) and on recovery. New **"Playout outputs" preflight row**: ok "N of N responding to the heartbeat", warn when no window is open but the show has video, **fail naming the dead output** with the fix instruction. Verified live with a real output page over a real BroadcastChannel: healthy 1/1 → froze the output's event loop 9 s → dot timeline `live→dead→live`, both toasts captured in order ("⚠ stopped responding" / "✓ recovered — re-synced"), show log holds the paired error/recovery entries for every cycle; preflight row verified in both ok and fail states (screenshots). Zero console errors; contract lint still green.
   *Learnings:* the Browser-pane preview coerces `window.open` into same-tab navigation, so output-popup testing there needs a stubbed window handle + a real output tab (the watchdog only reads `.closed` from the handle — everything else rode the real channel); real Chrome is unaffected. Background-tab timer throttling delays `setInterval`/`setTimeout` heavily — irrelevant on show day (controller and outputs are visible) but it will matter if Phase 2 item 3's snapshot cadence or any future watchdog assumes background timing. `slog` categories: used `output` for recovery (green-ish) and `error` for death — the show-log filter chips pick these up automatically.

2. ✅ **Undo/redo for rundown edits (Cmd/Ctrl+Z)** (medium, high) *(Done 2026-07-11.)*
   Every committed rundown diff now captures its inverse and a plain-language label in a bounded 50-entry history; Cmd/Ctrl+Z undoes and Cmd/Ctrl+Shift+Z redoes while the rundown is open (native undo remains untouched in inputs and textareas). Replays update the local rundown immediately, then generate a fresh batch through the existing transaction/alias pipeline so collaborators converge. Each transaction publishes its `rundownBatchId`; clients distinguish their own acknowledgements from remote batches and drop undo/redo entries whose row ids overlap a collaborator's edit. History resets when the tab changes sessions. Verified against the local Firestore emulator with two live operator clients: add propagated A→B, undo removed the row on both, redo restored it on both, and a remote cue edit invalidated the conflicting local entry (`Nothing to undo`) without reverting the collaborator's work. Both clients converged after the transaction retry expected under simultaneous writes; zero console errors.
   *Learnings:* order changes intentionally count as touching every id in the order, so a collaborator reorder invalidates row histories conservatively rather than replaying an inverse into a changed sequence. The session snapshot feature in item 3 should remain the recovery path for older/conflicting edits that are deliberately evicted from this fine-grained stack. The stale NOW/NEXT label can briefly retain a just-undone row while still in Build (the rundown and Firestore state are already correct); later live-card render work should treat local batch replays like remote rundown renders.

### The flows (owner's spec)
- **First visit:** user hits cueola.live, taps the **profile button on the front page** (or taps any app that needs a session). If they have no username yet, they choose "Create profile" → enter the **login code** the admin gave the class → enter **full name** → pick a **username**, **profile image** (reuse the avatar portal: initials / brand animals / upload), and **theme** (changeable later) → enter the **session codes** they've been given.
- **Every visit after:** entering any app asks for their **username** → prompts them to **select which session** from their saved list (new session codes addable inline). No password.
- **Admins:** an **admin code** works the same way — logging in with it registers the profile as an admin, so admins are consistently identified and can attach session codes as they're given. (This aligns with the existing `admins/global` roster + codeHash system in dashboard.html — the admin code becomes a special login code.)
- **The profile portal shows:** assigned position(s) per session, paperwork they need to do, notes they need to see (unseen count), To-Dos/checklist items assigned to them, and other action items. *(This absorbs the "My Tasks view" item — it lives in the portal.)*

### Decision — MADE (owner, 2026-07-11)
- **Ship Option A as the default with B as a per-session admin toggle:** entering an app = type username → pick session from the profile's saved list; admins can flip "require login code on entry" per session for extra friction. Session codes remain the fallback path for guests/one-off crew.
- **Hard requirement: easy backend management.** The admin experience is a first-class deliverable, not an afterthought: one dashboard panel to mint/revoke login codes, see who registered on which code, rename/merge/deactivate users, and bulk-attach sessions to a class roster. Every schema choice below should be judged by "can the owner administer this in one screen".

### Backend sketch (needs rules work — coordinate with Phase 1 item 2)
- `accessCodes/{code}`: `{ role: 'student'|'admin', label: 'Fall 2026 TV Production', active, createdBy, createdAt }` — created/revoked by admins from the dashboard.

- Rules: profile create requires the referenced access code doc to exist and be active (rules `get()`); writes shape-validated; no self-role-elevation (role copied from the code doc, not client-supplied). Honest limitation: with no auth, a determined user can still read/write more than they should until App Check + rules land — this layer is about *consistency*, not secrecy.
- App integration: join flows (`joinSession`, Planda Bear join, Flowmingo, Outrangutan session) get a username-first path; `session.userName` becomes the profile's fullName/username so presence, notes, mentions, roleAssignments, and seen-by all key off one canonical name. Migrate the local `cueola_profile` avatar into the cloud profile.
- Portal aggregation: position from `roleAssignments`, open items from notes checklists/assignees (cheap after the Phase 4 per-note migration; workable before it), unseen notes from the seen-by data.
- Dashboard: admin UI to mint/revoke login codes, see who registered with which code, and manage a class roster (rename, merge duplicate identities, attach sessions to users in bulk).

### Why this phase sits here
It needs Phase 1's rules/emulator groundwork (new collections must be in the same rules change), and Phase 4's collaboration features all get simpler once identity is canonical (mentions pool, assignees, seen-by, role chips all key off profiles instead of ad-hoc names).

## Phase 4 — Collaboration backbone
*The notes board becomes structurally sound, then socially sticky — now on top of real identities.*

1. ✅ **Migrate Production Notes to a per-note Firestore subcollection** (large, high) *(Done 2026-07-12.)*
   `sessions/{code}/notes/{noteId}` is now the primary store: all eight board mutations (post, reply, like, pin, edit, to-do done, checklist tick, thread delete) funnel through one dispatcher (`pbApplyNoteMutation`) that writes a single small doc — likes ride `arrayUnion`/`arrayRemove`, field edits are masked `updateDoc` patches with a not-found upsert, deletes remove the docs AND purge the ids from the legacy array so the read-both merge can never resurrect a deleted thread. Loads and a new **subcollection `onSnapshot` listener** merge legacy array ∪ subcollection (subcollection wins per id) and lazily backfill legacy-only notes, idempotently by id; the listener starts from both `setupFirestore` and `joinPresence`, so **hub/notes-only joins finally get live push** (the gap flagged in owner-correction #2). Notification seeding semantics preserved (first delivery per session = history, no toast; one shared known-ids set across both sources). **Production-safe before the rules deploy:** the first permission-denied drops the session to `legacy` mode — the entire old transaction pipeline (`writePlandaBearNotes` + `pbReconcileNotes`) is retained as the fallback and even replays the failed mutation, so nothing is lost mid-action; the next load probes again, making the owner's staged-rules deploy the self-retiring upgrade switch. Wire format is compact (empty/default fields omitted; `''` in a patch becomes `deleteField`) because the staged rules treat present-but-empty strings as invalid; `validNote` gained type checks (done/pinned bool, at/doneAt/editedAt int, doneBy/clientId/replyTo/avatar) and the suite ~12 cases covering the live patch shapes. The Phase 3 portal merges subcollection notes into its badge counts. Verified end-to-end on the emulator with two live clients (rundown operator + notes-only join): legacy seed → lazy backfill (REST-confirmed compact docs); post → **one new doc, array length unchanged**; like/edit/tick/delete all propagate A↔B live with zero console errors; the old-rules drill (notes subcollection denied) fell back to array writes and, after restoring the staged rules, one reload self-upgraded and backfilled the fallback-era note.
   *Learnings that change later items:* **(a) Backfill vs. delete race, found live:** the client's own latency-compensated listener snapshot fires mid-delete and its backfill resurrected the note from the stale legacy-array cache. Fix: an `_pbEverInSub` set — an id ever observed in the subcollection is never backfilled again (disappearance = deletion), plus legacy-cache bookkeeping before the delete awaits. Any future dual-store window (item 2's assignees, item 3's seen-by if they add stores) must reason about self-triggered snapshots. **(b)** Items 2 and 3 can now write per-note masked patches (`checklist`, future `seenBy` map) through the same dispatcher — no array rewrites, and `seenBy` per-person keys should be masked field paths to dodge whole-map races. **(c)** This release still reads + backfills the array; the NEXT release (after the owner's rules deploy has settled behind the cache-bust) should stop reading `preProNotes`, delete `writePlandaBearNotes`/`pbReconcileNotes` outright (they live only as the fallback engine now), and optionally blank the array field. **(d)** Session-history restore (Phase 2 item 3) still restores the ARRAY only: never-migrated notes reappear correctly via merge+backfill, but restores cannot roll back per-note edits, and a fresh client's backfill can resurrect a sub-deleted note that a restore re-added to the array — per-note snapshot hooks remain the Phase 2 item 3 learning's future refinement. **(e)** The `_disableNetwork` console drill did not gate writes against the emulator connection in this run (the setDoc reached the server) — the offline-queueing win of plain per-note writes rests on SDK semantics plus Phase 1's drill; the owner's real-Wi-Fi rehearsal is still the faithful test.

## Key risks (short list)
1. Firestore persistence changes write-failure semantics — the audit in Phase 1 item 1 *is* the item.
2. Rules/migration ordering — every new collection (files, **profiles, accessCodes**, notes) edits `firestore.rules` + emulator tests in the same commit; the owner deploy runbook keys off it.
3. **Identity layer is not auth** — with open rules and no passwords (university constraint), usernames are spoofable by a determined user until App Check + hardened rules deploy; login codes can leak and need cheap revocation (deactivate the code doc, existing profiles keep working). Present it as consistency + convenience, never as security.
4. **Username collisions & renames** — doc-id-as-username makes uniqueness easy but renames hard; decide early whether rename = new doc + tombstone pointer, and how mentions/assignments follow a rename.
5. Mixed-version clients during the notes migration — cache-bust script first, idempotent backfill, read-both window.
6. Service-worker staleness foot-gun on GitHub Pages — mechanical cache naming + tested update path.
7. Undo/redo vs. concurrent collaborators — inverse batches must ride the alias-resolving pipeline; two-client QA.
8. Two QLab bridges exist (`bridge.py` + `agent.js`) — pick one, retire the other.
9. Thin symbol weights can shimmer at small sizes/low DPI — pick Light vs Ultralight by eye at 11–13px before committing the whole set.
10. No-build rule under pressure (zip writer, SW, MIDI, identity flows) — stay dependency-free.
