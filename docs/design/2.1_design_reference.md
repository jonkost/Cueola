# Design standards

Design guidelines for this project, based on Apple's Human Interface
Guidelines (HIG) adapted for web. Apply these to every component and screen.
When a design decision isn't covered here, resolve it by asking which option
best serves the three core principles below.

## Core principles

Everything in this document descends from three ideas. When in doubt, defer to these.

1. **Clarity** — Text is legible at every size, icons are precise, and every
   element earns its place. If a control doesn't look interactive, it has failed.
2. **Deference** — The interface stays out of the way; the user's content is the
   star. Chrome recedes. Never stack a nav bar, toolbar, tab bar, and floating
   button all competing on one screen.
3. **Depth** — Communicate hierarchy and relationships through layering and
   motion, not decoration. Use it purposefully, never for its own sake.

**The through-line: the interface must never compete with the content.**

## Layout & structure

- Build everything on an **8px spacing grid** with 4px subdivisions. All padding,
  margins, gaps, and sizing should be multiples of 8 (or 4 for fine tuning).
  This single habit does the most to make a layout feel right.
- Design **adaptively**, not for one fixed width. Use genuine responsive layout:
  the visual hierarchy stays constant while the layout reflows across screen sizes.
- Respect **safe areas** — keep content clear of viewport edges and any system UI.
  Use `env(safe-area-inset-*)` where relevant.
- **Group related controls** into clean sections separated by generous whitespace.
  Model: Apple's Settings app — grouped lists, consistent type, nothing fighting
  for attention. One screen should teach the user what to do at a glance.
- Prefer **whitespace over dividers** to separate content. Reach for borders only
  when grouping genuinely needs a hairline.

### Spacing scale

Use these tokens; don't invent one-off values.

```
--space-1: 4px;
--space-2: 8px;
--space-3: 12px;
--space-4: 16px;
--space-5: 24px;
--space-6: 32px;
--space-7: 48px;
--space-8: 64px;
```

## Navigation

- Favor **flat navigation** over deep hierarchy.
- Use a **tab bar** for primary, top-level sections. Keep it persistent and
  always visible (except within a modal).
- Use a **top bar** for the current screen's title plus contextual controls that
  act on the content below it. Back action lives top-left.
- Put standard controls where users expect them. Don't reinvent navigation.
- Use **modals sparingly** — overuse causes "modal fatigue." Prefer hierarchical
  navigation when content relationships justify it. Every modal needs clear
  **Done/confirm** and **Cancel** actions, and a casual dismissal must never
  destroy unsaved work.

## Typography

- One neutral **system typeface**. On web:
  ```css
  font-family: system-ui, -apple-system, "SF Pro", sans-serif;
  --font-mono: "SF Mono", ui-monospace, monospace;
  ```
- Use a defined **type scale**. Reference values (default size):

  | Role         | Size |
  |--------------|------|
  | Large title  | 34   |
  | Title 1      | 28   |
  | Title 2      | 22   |
  | Title 3      | 20   |
  | Headline     | 17   |
  | Body         | 17   |
  | Callout      | 16   |
  | Subhead      | 15   |
  | Footnote     | 13   |
  | Caption      | 12   |

- Body sits at **17**. Build hierarchy with size and weight, not decoration.
- Use **two weights** for most work (regular + medium/semibold). Avoid heavy stacks.
- Support **text scaling**: use `rem` units and respect the user's font-size
  setting. Don't lock everything to fixed `px`.

## Color

- Use a **limited, cohesive palette**. Let color carry meaning, not just style.
- **Never use one color to mean two different things**, and never rely on color as
  the *only* signal (accessibility) — pair it with text, icon, or shape.
- Use **semantic roles**:
  - Blue — primary / interactive actions
  - Red — destructive / errors
  - Green — success
  - Gray scale — neutral text, borders, backgrounds
- Keep primary, secondary, and disabled states **visually distinct**.
- Design for **light and dark mode from the start**. Drive all color through
  semantic CSS custom properties with a dark-mode override, so colors adapt
  automatically. Never hardcode hex values in components.

```css
:root {
  --color-accent: #007aff;
  --color-danger: #ff3b30;
  --color-success: #34c759;
  --text-primary: #000;
  --text-secondary: #3c3c43;
  --surface: #fff;
  --border: rgba(0,0,0,0.1);
}
@media (prefers-color-scheme: dark) {
  :root {
    --text-primary: #fff;
    --text-secondary: #ebebf5;
    --surface: #1c1c1e;
    --border: rgba(255,255,255,0.15);
  }
}
```

## Sizing, targets & accessibility

These are non-negotiable.

- **Minimum interactive target: 44×44px.** Don't crowd targets together.
- **Contrast: at least 4.5:1** for body text against its background (3:1 for
  large text). Verify in both light and dark mode.
- **Support text scaling** (see Typography).
- Provide the **full experience to assistive tech**: semantic HTML, proper labels
  (`aria-label` on icon-only controls), logical heading order, full keyboard
  navigation with visible focus states.
- Honor user preferences: `prefers-reduced-motion`, `prefers-contrast`,
  `prefers-color-scheme`.
- Treat accessibility as **structural**, decided up front — not a polish pass.

## Motion

- Motion should feel physical and purposeful — it clarifies relationships and
  maintains continuity between states.
- Keep it smooth and restrained. Disable non-essential animation entirely under
  `prefers-reduced-motion: reduce`.

## Anti-patterns to avoid

- Multiple competing chrome elements on one screen (nav + toolbar + tab bar +
  floating button all at once).
- Arbitrary spacing values off the 8px grid.
- Color as the sole carrier of meaning.
- Fixed pixel type that ignores user scaling.
- Tiny or crowded tap targets.
- Decoration that competes with content.
